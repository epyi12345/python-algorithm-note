    시간복잡도와 메모리복잡도

    이 개념은 무엇인가

    복잡도 체크는 코드를 작성하기 전에 실행 시간과 메모리 사용량을 미리 추정하는 습관입니다. 제약 조건에 비해 너무 느리거나 너무 큰 풀이를 초기에 걸러내는 데 도움이 됩니다.

    시간복잡도(time complexity)는 연산 횟수가 입력 크기에 따라 어떻게 증가하는지 나타냅니다. 메모리복잡도(space complexity)는 알고리즘이 추가로 사용하는 저장 공간이 얼마나 커지는지 나타냅니다.

    언제 사용하는가

    문제를 읽은 직후, 특히 다음 신호가 보이면 바로 확인합니다.

    • N, Q, N * M 같은 큰 제약 조건이 있다.
    • 같은 데이터에 대해 반복 질의가 들어온다.
    • 큰 입력에 중첩 반복문을 쓰려고 한다.
    • 큰 배열, 격자(grid), dict, set, 큐(queue), 스택(stack), DP 테이블을 만든다.
    • Python 재귀가 깊어질 수 있다.

    핵심 아이디어

    먼저 가장 큰 입력 크기를 읽고, 최악의 경우 연산 횟수와 가장 큰 자료구조를 추정합니다.

    예를 들어:

    • N은 보통 1차원 입력 크기입니다.
    • Q는 보통 질의 수입니다.
    • N * M은 격자(grid)의 모든 칸 수를 의미하는 경우가 많습니다.
    • N과 Q가 모두 크면 O(NQ)는 위험합니다.

    예제 입력에서 맞는 풀이도 최악의 경우 입력에서는 시간 초과나 메모리 초과가 날 수 있습니다.

    시간복잡도 체크

    제약 조건 읽기

    코딩 전에 중요한 최댓값을 적어 봅니다.

    N <= 100000
Q <= 100000

    그다음 풀이의 반복 구조를 추정합니다.

    질의 하나당 직접 계산: O(N)
질의 수: Q
전체: O(NQ)

    N = 100000, Q = 100000이면 O(NQ)는 대략 다음 수준입니다.

    100000 * 100000 = 10000000000

    일반적인 코딩테스트 제한 시간에서는 너무 큽니다.

    자주 쓰는 판단 기준

    정확한 기준은 언어, 채점 서버, 제한 시간에 따라 달라지지만, 초반 판단에는 다음 기준이 유용합니다.

    제약 조건 형태 보통 가능한 접근 위험한 접근
    작은 N 예: N <= 100 중첩 반복문도 가능할 수 있음 너무 이른 과최적화
    N <= 1000 O(N^2) 가능 범위 O(N^3)는 위험할 수 있음
    N <= 100000 O(N) 또는 O(N log N) O(N^2)
    질의 Q가 많음 전처리 + 빠른 질의 처리 질의마다 처음부터 다시 계산
    격자 N * M 각 칸을 한 번 방문 전체 격자를 여러 번 반복 스캔

    예시

    접근 시간복잡도 이유
    질의마다 직접 구간 합산 O(NQ) 각 질의가 최대 N개 값을 훑을 수 있음
    누적합(prefix sum) O(N + Q) 한 번 전처리하고 각 질의를 O(1)에 처리
    정렬 O(N log N) 일반적인 비교 정렬 비용
    격자 BFS O(N * M) 각 격자 칸을 최대 한 번 방문

    메모리복잡도 체크

    메모리복잡도는 동시에 존재하는 큰 자료구조가 무엇인지를 묻는 과정입니다.

    문제에서 요구하지 않는 한 정확한 바이트 수를 단정할 필요는 없습니다. Python 런타임과 실행 환경에 따라 달라질 수 있으므로, 실전에서는 Big-O 기준의 메모리 추정이 더 안전합니다.

    자주 쓰는 추정은 다음과 같습니다.

    자료구조 메모리복잡도
    입력 리스트/배열 O(N)
    누적합(prefix sum) 배열 O(N)
    2차원 격자(grid) O(N * M)
    2차원 방문 배열 O(N * M)
    dict 또는 set O(number of unique keys)
    큐(queue) 또는 스택(stack) 최악의 경우 O(N) 또는 O(N * M)
    DP 테이블 상태 수에 따라 결정
    재귀 호출 스택 재귀 깊이에 따라 결정

    리스트 슬라이싱과 복사도 비용이다

    Python에서 리스트 슬라이싱은 새 리스트를 만듭니다.

    part = arr[l:r]

    이 코드가 반복문 안에 있으면 시간과 메모리 사용량이 함께 증가할 수 있습니다. 구간 합 문제에서는 반복 슬라이싱보다 누적합(prefix sum)을 우선 고려합니다.

    출력은 스트리밍할 수 있는지 확인하기

    문제가 즉시 출력해도 되는 형태라면 모든 중간 결과를 저장할 필요가 없을 수 있습니다. 속도를 위해 batched output을 쓰는 경우도 있지만, 큰 결과 리스트를 무조건 들고 있는 습관은 피합니다.

    코딩 전 실전 체크리스트

    구현 전에 다음 질문을 확인합니다.

    • 가장 큰 입력 크기는 얼마인가?
    • 중첩 반복문이 몇 겹인가?
    • 반복 질의가 있는가?
    • 전처리할 수 있는가?
    • 어떤 큰 배열, set, dict, 큐(queue), 스택(stack), 방문 배열을 만들 것인가?
    • 메모리가 N, N * M, 또는 그보다 큰 값에 비례하는가?
    • 실수로 리스트나 슬라이스를 복사하고 있지 않은가?
    • 모든 데이터를 저장하지 않고 스트리밍/순차 처리할 수 있는가?
    • Python 재귀 깊이가 안전한가?

    흔한 메모리 실수

    큐(queue)에 list.pop(0) 사용

    list.pop(0)은 앞 원소를 제거한 뒤 나머지를 당겨야 해서 큐(queue) 용도로 비효율적입니다. collections.deque와 popleft()를 사용합니다.

    반복문 안에서 슬라이스 생성

    for l, r in queries:
    total = sum(arr[l:r])

    각 질의마다 새 리스트 슬라이스가 생겨 시간과 메모리를 모두 낭비할 수 있습니다. 인덱스 기반 처리나 전처리를 고려합니다.

    모든 질의 결과를 불필요하게 저장

    바로 출력할 수 있다면 모든 답을 리스트에 저장하지 않아도 됩니다. 다만 출력 성능을 위해 batched output이 필요한 경우에는 저장량을 의식해서 관리합니다.

    같은 리스트를 여러 번 불필요하게 복사

    list(arr), arr[:], 반복 변환은 큰 입력을 그대로 복제할 수 있습니다. 정말 필요한 경우가 아니라면 하나의 표현을 유지합니다.

    재귀 깊이 고려 없이 DFS 작성

    재귀 DFS는 깊이만큼 호출 스택을 사용합니다. Python에서는 깊은 재귀가 RecursionError를 만들거나 스택 사용량을 키울 수 있습니다. 깊이가 클 수 있으면 반복형 DFS를 고려합니다.

    너무 큰 2차원 방문 배열 생성

    N * M 격자의 방문 배열은 O(N * M)입니다. 일반적인 격자 BFS에서는 자연스럽지만, 제약 조건이 매우 큰 희소 공간이라면 좌표 set으로 방문 처리를 하는 방식이 더 적절할 수 있습니다.

    패턴별 복잡도 표

    패턴 시간 메모리 메모
    단일 반복문 O(N) O(1) 또는 O(N) 결과 저장 여부에 따라 달라짐
    정렬 O(N log N) 구현/출력 방식에 따라 다름 sorted()는 새 리스트를 반환하고 .sort()는 제자리 정렬
    누적합(prefix sum) O(N + Q) O(N) 반복 구간 질의에 적합
    투 포인터(two pointers) O(N) 보통 O(1) 연속 구간 문제에 자주 사용
    단조 스택(monotonic stack) O(N) O(N) 각 인덱스가 최대 한 번 push/pop
    격자 BFS O(N * M) O(N * M) 큐(queue)와 방문 배열이 격자 크기만큼 커질 수 있음
    이분 탐색 O(log answer * check_cost) 검사 함수에 따라 다름 정렬된 공간 또는 정답 후보 탐색에 사용
    DP 상태 수에 따라 결정 상태 수에 따라 결정 코딩 전에 상태(state)를 정의해야 함

    안전한 패턴

    1. N, Q, 격자 크기, 값 범위를 확인합니다.
    2. 반복문과 반복 계산으로 시간복잡도(time complexity)를 추정합니다.
    3. 저장하는 자료구조로 메모리복잡도(space complexity)를 추정합니다.
    4. 명확히 너무 큰 접근은 버립니다.
    5. 제약 조건에 맞는 패턴을 선택합니다.

    예시:

    구간 합 질의가 많다 -> 누적합(prefix sum)
양수 연속 구간을 조절한다 -> 투 포인터(two pointers) / 슬라이딩 윈도우(sliding window)
다음 큰 값을 찾는다 -> 단조 스택(monotonic stack)
가중치 없는 격자 최단거리 -> BFS
가능한 최솟값을 찾는다 -> 이분 탐색(binary search) / 매개변수 탐색(parametric search)
중복 부분 문제가 반복된다 -> DP
